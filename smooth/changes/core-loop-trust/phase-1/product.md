# 产品需求

## 阶段定位

`core-loop-trust` 的整体目标是让用户能够相信 Orbit Wars 对比赛类型、模拟结果、发布资格和比赛进度的表达。Phase 1 只修复真实用户链路中的可信度断点，形成“准备策略 → 私人模拟 → 验证发布 → 创建比赛 → 追踪状态 → 查看回放 → 返回策略实验室”的可恢复闭环。后续阶段再讨论模拟分享、统计型发布门槛、多舰队和更完整的导航体验。

## 背景

真实用户链路已跑通，但核心循环中存在以下问题：

- `PublicCompetition` 历史将 training 和 strategy simulation 硬编码显示为 `RANKED`，私人候选策略模拟也会进入公开历史。
- strategy simulation 尚处于 `QUEUED` 时发布按钮已经解锁；刷新页面或切换语言后，当前 simulation 状态丢失。
- Arena 创建比赛后，“查看执行状态”进入 Command Center 的 Agent Key 页面，用户无法从比赛的 `queued`、`running`、`finished` 状态一路追踪到 replay。
- replay 进入 strategy lab 时虽然携带 `fromReplay`，界面没有显示来源战果、回放高光或返回入口。
- 中文 start 页仍有英文主标题；已有舰队的用户仍被引导进入新建舰队流程。
- 菜单中的“退出”含义不够明确，且在未登录时仍然显示。

这些问题不会阻止所有操作完成，却会让用户无法判断当前行为是否公开、策略是否真的通过验证、比赛是否仍在执行，以及优化动作来自哪一场战斗。Phase 1 优先恢复这条链路的可解释性和连续性。

## 根因

1. **展示没有统一服从权威业务状态。** 比赛类型由页面场景或硬编码文案推断，隐私与公开历史也未严格绑定。
2. **模拟被当作页面临时状态。** simulation 与候选策略的关联、执行进度和验证结果没有成为可恢复的用户任务，导致刷新、切语言和重新进入页面后上下文断裂。
3. **发布资格判断过早。** 前端看到“已发起模拟”就允许发布，没有等待关联 simulation 完成，也没有在发布时以最新状态再次确认资格。
4. **比赛创建与比赛追踪脱节。** 创建成功后的行动入口没有携带具体比赛上下文，落入面向 Agent 接入的通用 Command Center，而不是玩家需要的比赛状态页。
5. **回放来源仅存在于参数中。** `fromReplay` 没有转化为可见、可返回的策略优化上下文。
6. **本地化、鉴权和舰队状态没有共同驱动入口。** 页面文案与 CTA 仍按静态默认状态展示。

## 产品目标

- 用户在任何历史和详情中都能看到比赛的真实 mode，不会把训练或模拟误认成排位赛。
- 私人 strategy simulation 默认只对发起者可见，不进入任何公开比赛历史或公开发现入口。
- 用户只有在关联 simulation 已完成且验证通过时才能发布候选策略，并能理解未解锁原因。
- simulation 在刷新、切换语言和重新进入后仍可恢复；进行中的状态会继续更新。
- 用户创建 Arena 比赛后可以直接查看该比赛状态，并从排队、运行一路到达回放。
- 从 replay 进入 strategy lab 时，用户能看见最小但明确的来源上下文，并能返回原 replay。
- start 页和账户菜单在中英文、登录状态与舰队状态下给出准确入口。

## 用户旅程

### 旅程一：模拟并发布候选策略

1. 用户在 strategy lab 创建或修改候选策略并发起 strategy simulation。
2. simulation 默认标记为私人；用户看到与该候选策略关联的状态。
3. simulation 处于 `QUEUED` 或 `RUNNING` 时，页面持续更新状态，发布入口保持锁定并说明原因。
4. 用户刷新页面、切换语言或稍后重新进入，页面恢复同一个关联 simulation 的最新状态。
5. simulation 进入 `COMPLETED` 后，系统展示验证结果：
   - 验证通过：发布入口解锁。
   - 验证未通过：发布入口保持锁定，并显示可行动的失败原因或重新模拟入口。
6. 用户提交发布时，系统再次依据该候选策略关联 simulation 的最新权威状态与验证结果确认资格，避免使用过期页面状态。

### 旅程二：开赛并追踪到回放

1. 用户在 Arena 创建比赛。
2. 创建成功后，主行动入口直接进入这场比赛的状态视图，不要求 Agent Key，也不进入通用接入页面。
3. 状态视图围绕同一个比赛标识持续展示 `QUEUED`、`RUNNING`、`FINISHED` 等状态。
4. 比赛结束且 replay 可用后，主行动入口变为“查看回放 / Watch replay”。
5. 若比赛已结束但 replay 尚未可用，页面明确显示“回放生成中 / Replay processing”并继续更新，而不是给出失效入口。
6. 失败或取消时，页面显示终态与可理解原因，并提供返回 Arena 的入口。

### 旅程三：从回放继续优化

1. 用户从 replay 进入 strategy lab。
2. strategy lab 顶部显示来源战斗卡片，至少包含战果、一个最小高光和“返回回放”入口。
3. 用户在保留来源上下文的情况下开始新一轮策略调整与私人模拟。
4. 返回入口回到原 replay，而不是回放列表或无关页面。

### 旅程四：从 start 页继续游戏

1. 未登录用户看到登录或开始入口，不看到“退出登录”。
2. 已登录但没有舰队的用户看到“创建第一支舰队 / Create your first fleet”。
3. 已登录且已有可用舰队的用户看到单一主 CTA“进入竞技场 / Enter Arena”，不再重复进入新建流程。
4. 已登录但舰队尚未达到可用状态的用户看到“继续配置舰队 / Continue fleet setup”。

## 范围

### Phase 1 纳入

- 公开历史与相关比赛展示按权威 mode 显示正确类型。
- 私人 strategy simulation 默认不进入公开历史、公开列表或公开发现入口。
- 候选策略与其验证 simulation 的关联状态可恢复。
- 发布资格严格要求关联 simulation 为 `COMPLETED` 且验证通过。
- simulation 进行态刷新、页面重载、语言切换和重新进入后的恢复。
- Arena 创建比赛后进入比赛专属、可轮询的状态与 replay 路径。
- replay → strategy lab 的最小来源战果、高光与返回入口。
- start 页明显的中英文混用修复，以及按登录/舰队状态显示正确主 CTA。
- 菜单“退出”调整为“退出登录 / Sign out”，未登录时隐藏。

### Phase 1 不纳入

- 多舰队创建、切换、选择、编队管理或多舰队专属首页逻辑。
- 付费、订阅、付费解锁或商业化门槛。
- 基于多场样本、胜率、置信区间或复杂评分的发布门槛。
- 私人 simulation 的公开分享、链接分享、协作权限、可撤销分享或分享发现页。
- 移动导航重构、一级导航信息架构调整或全站视觉改版。
- 新的回放高光识别算法、完整赛后分析系统或复杂策略对比工具。
- Command Center 的 Agent 接入流程重构；本期只确保比赛状态入口不再误入该流程。

## 详细状态规则

### 1. 比赛 mode 与公开性

比赛类型展示必须读取该比赛或 competition 的权威 mode 字段，不得根据入口页面、历史列表类型、是否计分或前端默认值推断。

| 权威 mode | 中文显示 | 英文显示 | 是否可进入公开比赛历史 |
| --- | --- | --- | --- |
| `RANKED` | 排位赛 | Ranked | 是，前提是该比赛本身允许公开 |
| `TRAINING` | 训练赛 | Training | 仅在现有产品规则明确允许公开时展示，但不得显示为排位赛 |
| `STRATEGY_SIMULATION` | 策略模拟 | Strategy Simulation | 默认否；私人模拟仅发起者可见 |
| 缺失或未知 | 未知模式 | Unknown mode | 否；不得回退为排位赛 |

- mode 标签在公开历史、用户可见历史、比赛状态、replay 和 replay 来源上下文中保持一致。
- “私人”是可见性约束，不是 mode 的别名；私人 strategy simulation 仍显示其真实 mode。
- 私人 simulation 可以出现在发起者自己的 strategy lab 记录或恢复上下文中，但不得进入公共历史、公共榜单动态、公共搜索或其他用户可访问的比赛列表。
- Phase 1 不提供把私人 simulation 改为公开的入口。

### 2. simulation 生命周期与恢复

页面必须展示服务端记录的最新状态。Phase 1 使用以下用户可见规则；如果底层存在更多状态，应归并到最接近的用户语义，不得误显示为已完成。

| simulation 状态 | 用户可见含义 | 页面行为 | 发布资格 |
| --- | --- | --- | --- |
| `QUEUED` | 已进入队列，等待执行 | 自动刷新；允许离开后恢复 | 锁定 |
| `RUNNING` | 正在执行 | 自动刷新；允许离开后恢复 | 锁定 |
| `COMPLETED` + 验证通过 | 模拟完成，候选策略验证通过 | 展示结果与通过状态 | 解锁 |
| `COMPLETED` + 验证未通过 | 模拟完成，但候选策略未通过验证 | 展示失败原因与重新模拟入口 | 锁定 |
| `FAILED` | 模拟执行失败 | 展示失败原因与重试入口 | 锁定 |
| `CANCELLED`（如存在） | 模拟已取消 | 展示终态与重新模拟入口 | 锁定 |
| 状态未知、关联缺失或暂时不可达 | 无法确认模拟结果 | 展示“状态待确认”，允许重试刷新 | 锁定 |

- 每次候选策略发布资格只由与该候选版本明确关联的 simulation 决定；其他策略或旧版本的成功 simulation 不得解锁当前候选策略。
- 页面首次进入、浏览器刷新、语言切换、从其他页面返回及恢复登录会话时，都重新取得关联 simulation 的最新状态。
- `QUEUED` 和 `RUNNING` 状态在页面停留期间持续更新；进入终态后停止进行态刷新。
- 切换语言只能改变文案，不得创建新 simulation、清空关联、回退状态或解锁发布。
- 发布按钮旁应提供简短原因，例如“等待模拟完成”“验证未通过”或“无法确认模拟状态”，不能只呈现不可点击状态。
- 用户点击发布时必须以最新权威状态再次确认：只有关联 simulation 为 `COMPLETED` 且验证通过才允许成功发布。页面曾经解锁不构成发布凭证。
- Phase 1 的“验证通过”沿用现有单场验证规则，不增加胜率或多场统计门槛。

### 3. Arena 比赛状态与 replay

创建成功后的状态页必须绑定具体比赛，且在重新打开或刷新时仍能恢复该比赛的最新状态。

| 比赛状态 | 用户可见含义 | 主行动 |
| --- | --- | --- |
| `QUEUED` | 比赛已创建，等待执行 | 留在状态页并自动更新 |
| `RUNNING` | 比赛进行中 | 留在状态页并自动更新 |
| `FINISHED` 且 replay 可用 | 比赛结束 | 查看回放 |
| `FINISHED` 但 replay 未就绪 | 比赛结束，回放生成中 | 继续更新，回放就绪后自动提供入口 |
| `FAILED` | 比赛执行失败 | 查看原因并返回 Arena |
| `CANCELLED`（如存在） | 比赛已取消 | 返回 Arena |
| 状态未知或暂时不可达 | 暂时无法取得比赛状态 | 手动重试并保留比赛上下文 |

- 创建成功后的“查看执行状态 / View match status”必须进入比赛专属状态视图。
- 此旅程不得要求输入 Agent Key，也不得把 Agent Key 页面作为比赛状态的中转页。
- `QUEUED`、`RUNNING` 和 replay 处理中都应显示当前阶段与最近一次成功更新时间，让用户知道页面仍在工作。
- 状态更新失败不能永久锁死页面；用户可手动重试，恢复后继续沿同一比赛到达 replay。
- replay 可用后，无论用户一直停留在页面还是稍后重新进入，都能看到“查看回放 / Watch replay”。

### 4. replay 来源上下文

当 strategy lab 从有效的 replay 进入时，显示紧凑的来源战斗卡片：

- 来源战果：胜、负或平，并显示对手或比赛标识，避免只出现无上下文结果。
- 最小高光：至少一条简短高光。优先使用已有 replay 摘要或关键事件；若没有结构化高光，则以终局战果摘要作为高光，不要求 Phase 1 新增高光算法。
- 返回入口：“返回回放 / Back to replay”，精确返回来源 replay。
- mode：按同一权威 mode 规则显示，私人策略模拟不得被标成排位赛。

`fromReplay` 缺失、无效、无权限或对应 replay 不存在时，不展示伪造或残留的来源卡片；strategy lab 仍可作为普通入口使用，并给出不打断操作的来源不可用提示。

### 5. start 页与账户菜单

start 页主 CTA 由登录状态与当前舰队可用性共同决定：

| 用户状态 | 中文主 CTA | 英文主 CTA |
| --- | --- | --- |
| 未登录 | 开始游戏 | Start playing |
| 已登录，无舰队 | 创建第一支舰队 | Create your first fleet |
| 已登录，有可用舰队 | 进入竞技场 | Enter Arena |
| 已登录，仅有未完成或不可用舰队 | 继续配置舰队 | Continue fleet setup |

- Phase 1 只使用当前或默认舰队完成判断，不引入舰队选择器。
- 已有可用舰队的用户不得再次进入“新建第一支舰队”流程。
- 中文页面的主标题、主 CTA、辅助说明和账户菜单不得混入未约定的英文文案；英文页面同理。
- 已登录菜单使用“退出登录 / Sign out”；未登录菜单完全隐藏该项。

## 中英文与术语要求

| 概念 | 简体中文 | English |
| --- | --- | --- |
| Ranked | 排位赛 | Ranked |
| Training | 训练赛 | Training |
| Strategy Simulation | 策略模拟 | Strategy Simulation |
| Public Match History | 公开比赛历史 | Public Match History |
| Match Status | 比赛状态 | Match Status |
| Replay | 回放 | Replay |
| Strategy Lab | 策略实验室 | Strategy Lab |
| Sign out | 退出登录 | Sign out |

- 同一概念在历史、状态页、replay 和 strategy lab 中使用同一术语，不用 `RANKED`、Ranked Match、天梯赛等多个变体混杂展示。
- 状态文案应直接描述“正在发生什么”和“用户接下来能做什么”，避免使用 worker、artifact、polling 等内部实现词。
- 中文采用简洁、明确的产品语气；英文采用自然的 sentence case。专有产品名 `Orbit Wars`、`Arena`、`Command Center` 保持品牌写法。
- 切换语言后业务状态、当前比赛、当前 simulation、来源 replay 和按钮资格保持不变。

## 验收标准

1. 给定一条 training 比赛和一条 strategy simulation，公开或个人可见页面分别显示“训练赛 / Training”和“策略模拟 / Strategy Simulation”，均不显示为排位赛。
2. 给定未知或缺失 mode，页面显示“未知模式 / Unknown mode”，且该记录不进入公开历史，不回退为 `RANKED`。
3. 给定私人 strategy simulation，除发起者自己的恢复与记录场景外，公共历史、公共列表和其他用户视角均不可见。
4. 给定当前候选策略的关联 simulation 为 `QUEUED` 或 `RUNNING`，发布入口保持锁定并显示原因；刷新页面和切换中英文后状态与关联不丢失。
5. 给定关联 simulation 为 `COMPLETED` 且验证通过，发布入口才解锁；如果验证未通过、执行失败、状态未知或关联属于旧候选版本，发布仍被拒绝。
6. 给定页面曾显示可发布，但提交前服务端最新状态不再满足条件，发布不会成功，并显示最新原因。
7. 给定 Arena 比赛创建成功，“查看执行状态”直接打开该比赛状态，不出现 Agent Key 要求。
8. 给定比赛从 `QUEUED` 变为 `RUNNING` 再变为 `FINISHED`，状态页无需用户寻找其他入口即可持续更新；replay 可用后出现有效的“查看回放 / Watch replay”入口。
9. 给定比赛已结束但 replay 尚未就绪，页面显示回放生成中并继续更新；刷新或稍后重返后仍能到达同一 replay。
10. 给定从有效 replay 进入 strategy lab，页面显示来源战果、至少一条最小高光和精确返回原 replay 的入口；无效或无权限来源不会泄露数据或显示残留上下文。
11. 给定中文 start 页，主标题与关键操作无意外英文；给定英文 start 页，关键文案无意外中文。
12. 给定已登录且已有可用舰队，start 页主 CTA 为“进入竞技场 / Enter Arena”，不会进入新建第一支舰队流程。
13. 给定已登录但无舰队或仅有未完成舰队，分别显示创建第一支舰队或继续配置舰队的正确 CTA。
14. 给定未登录用户，菜单不显示退出项；给定已登录用户，菜单显示“退出登录 / Sign out”。
15. 在桌面和现有移动布局中，新增状态、双语文案和来源卡片可读且不溢出；本项不要求重构移动导航。

## 后续讨论项（不阻塞 Phase 1）

### 私人 simulation 分享

Phase 1 坚持默认私人且没有分享入口。未来若开放分享，需要先决定分享对象是只读 replay、完整 simulation 结果还是可复制策略，并明确链接权限、到期、撤销、被分享者身份和是否允许进入公开发现页。推荐先从可撤销的只读链接开始，避免“分享”隐式改变公开性。

### 多场胜率发布门槛

Phase 1 仅要求一个与当前候选版本关联、已完成且验证通过的 simulation。未来可讨论最少场次、对手覆盖、胜率或置信度，但门槛必须可解释，且要避免随机波动让发布资格反复变化。推荐在积累真实数据后再定义，不在本期预设复杂阈值。

### 已有舰队首页 CTA

Phase 1 采用单一主 CTA“进入竞技场 / Enter Arena”，优先让已有可用舰队的用户继续核心循环。后续可验证用户更常需要“继续上场”还是“优化策略”，再决定是否按最近任务动态调整主 CTA；动态入口不能让用户每次回首页都重新理解路径。

### 多舰队

Phase 1 仅依赖当前或默认舰队，不提供选择和管理。未来需要先定义默认舰队、参赛资格、跨舰队策略关系，以及从首页、Arena 和 strategy lab 切换舰队时如何保持上下文，再进入产品设计。

### 术语风格

Phase 1 采用直接、统一、可翻译的功能术语，以避免状态误解。后续可讨论更具世界观的品牌化命名，但必须保留清晰副标题或解释，并确保中文、英文与状态枚举之间存在稳定映射。
