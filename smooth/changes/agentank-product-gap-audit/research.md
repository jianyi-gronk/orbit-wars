# 前置调研

## 调研问题

- 截至 2026-07-18，Agentank 哪些产品机制让用户更容易理解并完成 Agent 游戏循环？
- Orbit Wars 已经具备哪些对应能力，差距主要位于后端能力还是用户旅程？
- 哪些机制值得转化为 Orbit Wars 的星战表达，哪些功能现在不应跟进？

## 已验证事实

### Agentank

- 首页用一句话明确“创建一个会自主战斗的坦克”，并把用户角色定义为教练、所有者和调优者，而不是直接操作者。
- 首页连续解释“创建坦克 → 交给 Agent → 观看回放并迭代”的三步循环，同时提供真实战场预览、规则说明和最新公开对局入口。
- 对局历史卡片直接展示胜负、双方、地图、结束原因、时间和 Heat，并提供“Exciting only”筛选。
- Agent Guide 给出一条完整可执行链路：读取上下文、模拟、发布、查询榜单/对手、发起真实对局、读取对局和回放。
- Agentank 同时提供面向人的回放 URL 与面向 Agent 的紧凑 JSON；Agent JSON 先返回摘要和关键事件，再按需提供事件流、帧切片和原始回放。
- 结束态支持重新开始、分享回放和导出视频；排行榜支持今日、本周、历史以及胜率、胜场、精彩度等排序。
- Agentank 还有 Cups、团队、商店、成就、邀请奖励、通知、社区评论、钱包和多种战场风格等扩展系统。

### Orbit Wars

- 首页已有四幕沉浸式叙事和“建立舰队 → 部署 Agent → 复盘进化”的三步表达，但首屏战场是装饰性雷达，预热 Agent 数字也是静态展示，不是实时战斗内容。
- 首页存在多个开始/历史/榜单入口，但没有一个贯穿全站、根据“未登录 / 未建舰队 / 未生成 Key / 未发布版本 / 可开战 / 刚结束比赛”自动变化的唯一下一步。
- StartFlow 已支持登录门、舰队身份、策略倾向和默认/Kaggle 模板；Command Center 已支持段位、策略版本、Agent Key、版本切换和进入竞技场。
- Agent API 已支持读取舰队、版本、对手和最近比赛，并支持候选模拟、版本发布和真实挑战；主要闭环能力已经存在。
- 公共榜单已支持今日、本周和历史周期；历史页已支持精彩筛选、控制标识、真实回放、版本归属和 rating delta。
- 回放 API 已支持 compact 摘要、关键事件、事实型胜因、评分变化、分段帧和完整工件；回放 UI 已支持时间轴、事件点、播放控制、兵力轨迹和事实型胜因。
- 当前历史卡片把 schema 版本、帧数、工件大小和保存时间放在较强视觉层级；这些信息可验证但不是普通观众决定是否观看的主要依据。
- 当前 Agent Guide 页面解释了完整循环与安全边界，但 Command Center 只提供单独“复制密钥”，没有把 Key、Guide URL 和下一条明确指令组合成一次可复制的 Agent 交接包。

## 当前差距与机会

### P0：优先解决完整循环

1. **全站状态感知的唯一主行动**
   - 将当前分散的 CTA 收敛成一个持续可见的“下一任务”入口。
   - 状态依次为：登录 → 创建舰队 → 复制 Agent 交接包 → 等待/确认 ready 版本 → 开始比赛 → 查看刚结束回放 → 继续迭代。
   - 这比增加更多导航更能解决新用户“不知道下一步做什么”。

2. **首页展示真实战斗，而不是装饰性雷达**
   - 用一场最近精彩回放的短循环、双方名称、段位、当前 step 和结果替换或增强首屏雷达。
   - 下方增加 3～5 场最新公开对局，卡片直接进入回放。
   - 首页因此既解释玩法，也能立即证明平台确实有人/Agent 在战斗。

3. **回放结束后直接进入 Agent 迭代**
   - 增加“一键交给 Agent 分析”：复制 compact replay URL、比赛结果、关键事件与一条分析提示词。
   - 同时提供“复制人类回放链接”和“复制 Agent 数据链接”，不要求用户理解分段协议。
   - 回放结束态明确给出下一步：分析 → 发布新版本 → 再战。

4. **Command Center 增加 Agent 交接包**
   - 生成 Key 时同时提供一次性“复制全部”：Guide URL、Key、API Base URL 和推荐首条指令。
   - 已有舰队但无 Key、无 ready 版本或刚完成比赛时，Command Center 的主卡片应显示不同下一步，而不是始终“让 Agent 出战”。

### P1：提升观看、传播和成长反馈

5. **历史卡片观众化**
   - 强化胜负标记、双方段位、rating delta、结束原因、关键事件和精彩度。
   - schema、工件大小和保存信息折叠到“验证详情”，避免技术元数据抢占主层级。

6. **回放分享与高光**
   - 提供复制链接、分享卡片；后续再考虑导出短视频。
   - 用现有关键事件生成 2～4 个可跳转高光点，不必先建设完整视频管线。

7. **榜单增加排序与成长目标**
   - 在已有周期筛选上补充总分、胜率、胜场、近期表现/精彩度排序。
   - Command Center 和公开舰队页显示距离下一段位还差多少分，并推荐合适对手。

8. **策略版本人格与归因**
   - 强化 submittedBy、版本备注、最近战绩和代表回放，让同一舰队的版本迭代有可读故事。
   - 不需要复制 TankBook，可先从“Agent 赛后一句话复盘”开始。

### P2：有活跃度后再建设

- 通知和每日战报。
- Agent 赛后评论、舰队留言墙和社区关系。
- 杯赛、团队赛与官方赛事。
- 成就、邀请奖励和装饰性收集系统。

## 暂不建议跟进

- **更多一级导航**：Agentank 功能多，因此导航也更密；Orbit Wars 已明确偏好少量主入口，不应照搬。
- **商店、钱包、Mint、奖励经济**：当前核心闭环和活跃度尚未验证，容易稀释产品主线。
- **Human 手动玩法和网页代码编辑器**：当前版本已决定 Agent-only，保留技术口子即可。
- **多模式、团队和 Cups**：应在 1v1 Agent 循环具备稳定留存后再扩展。
- **多种战场皮肤**：先把唯一战场的可读性、轨迹和回放体验打磨好。

## 风险与约束

- 首页真实回放必须处理无数据、加载失败、低性能设备和 reduced-motion，不能让首屏依赖一个脆弱的重型 Canvas。
- Agent Key 明文只出现一次；“复制交接包”必须在生成时完成，后续不能重新展示密钥。
- Agent 分析入口不得把密钥写入提示词、URL、回放或日志；公开 compact replay 本身不需要 Key。
- “精彩度”如果没有可解释评分，容易变成噱头。可先使用 featured + 关键事件数量，之后再定义 Heat。
- 当前首页四幕信息已经较多，加入真实战斗时需要替换装饰模块，而不是继续叠加第五幕。

## 对产品讨论的启发

- 下一轮需求最适合围绕一个主题收敛：**从任意用户状态到下一场可复盘对局的单一路径**。
- 推荐第一阶段只做三件事：状态感知主行动、首页真实回放预览、回放结束后的 Agent 分析交接。
- 这三项大多复用现有 API 与回放能力，能比新增游戏模式更快提升“这是一个真的能玩、能迭代的 Agent 游戏”的感受。
