# 踩坑记录

## 登录渠道不能从参考项目直接推断

- 症状：最初根据“韭见”的完整能力实现了邮箱注册界面，随后用户明确首发只支持 GitHub。
- 根因：参考产品包含 Email、GitHub 和 Google，但“可以参考”不等于首发渠道已经确认。
- 如何被发现：用户在浏览器验收前明确纠正为“先不开启邮箱登录，只支持 GitHub”。
- 修复 / 预防：GitHub 改为唯一可见入口；邮箱后端保留在默认关闭的 feature flag 后，API 与 UI 都不可用。
- 这能改进 harness 吗：是 — 产品分析阶段应把“首发身份提供方”列为显式决策，不根据参考产品自动扩展范围。

## 登录能力正确不代表入口形态正确

- 症状：GitHub 单入口、双语与回调均已完成，但登录被设计成占据整页的大型认证舞台，打断了用户当前任务。
- 根因：把“独立认证页面或全屏任务面板”都视作可接受方案，没有先锁定登录是主任务还是辅助动作。
- 如何被发现：用户对线上认证页明确要求参考“韭见”，改为登录弹窗即可。
- 修复 / 预防：首发 GitHub 登录由全局页头托管轻量弹窗；受保护任务复用同一入口，旧认证 URL 只兼容唤起，不维护第二套页面。
- 这能改进 harness 吗：是 — 认证产品分析必须显式确认入口形态、是否保留当前上下文，以及旧深链兼容策略。
