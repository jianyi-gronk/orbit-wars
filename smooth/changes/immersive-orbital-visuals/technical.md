# 技术设计

## 架构总览

```text
HomeExperience
├── DOM 内容层：标题、按钮、导航、四幕内容（现有）
├── CSS 降级层：星点、网格、雷达、信标（现有）
└── OrbitalWorld：增强画布（新增，aria-hidden）
    ├── activeScene ──> 镜头 / 世界目标状态
    ├── pointer ──────> 轻量视差目标
    ├── reducedMotion -> 静态渲染
    └── WebGL failure -> 静默退出，CSS 层继续工作
```

`HomeExperience` 继续拥有页面交互和场景状态，增强画布只消费状态，不反向控制 DOM，也不进入业务数据链路。

## 功能列表

1. 首页共享轨道世界
2. 四幕世界状态过渡
3. 性能、动效偏好与失败降级
4. 视觉层级与响应式整合
5. 自动化与浏览器验证

## 共享基础

### 状态契约

```ts
type OrbitalWorldProps = {
  activeScene: number;
  reducedMotion: boolean;
  pointer: { x: number; y: number };
};
```

场景索引沿用首页现有的 `0..3`。指针值为 `-1..1` 的归一化坐标，不直接传 DOM 事件，避免渲染组件依赖页面结构。

### 层级契约

从后向前依次为：页面底色 → CSS 星场降级 → Three.js 画布 → 场景装饰 → DOM 内容 → 固定导航与交互反馈。增强画布必须 `pointer-events: none`、`aria-hidden="true"`，不能成为焦点目标。

### 方案比较

| 方案 | 结果质量 | 成本与风险 | 结论 |
| --- | --- | --- | --- |
| 继续只用 CSS | 包体最小，但空间连续性和镜头变化有限 | 实现低，难达到目标 | 不选 |
| 复用 PixiJS | 粒子和 2D 合成成熟 | 透视轨道、球体光照需要自行模拟 | 不选 |
| 首页动态加载 Three.js | 适合程序化 3D、镜头和轨道空间 | 增加异步包与生命周期管理 | 采用 |
| HTML-in-Canvas 实验 API | DOM 与 WebGL 融合最直接 | 浏览器兼容性不可接受 | 仅作灵感 |

最重要的考量是增强层必须可独立失败。选择 Three.js 意味着接受首页首次加载一个额外异步模块，但不让它阻塞 DOM 首屏。

## 功能：首页共享轨道世界

### 涉及文件

- 新增 `apps/web/components/product/OrbitalWorld.tsx`
- 修改 `apps/web/components/product/HomeExperience.tsx`
- 修改 `apps/web/package.json` 与工作区锁文件

### 类型与接口

`OrbitalWorld` 接收共享状态契约，不持有场景导航状态。Three.js 使用客户端动态 `import()`，避免进入服务端渲染路径并让 DOM 首屏先完成。

### 场景对象

- 低密度远景星粒子：提供纵深，不承载交互。
- 程序化星球：球体、线框经纬层和背光层组成，不使用外部贴图。
- 多组椭圆轨道：不同倾角与透明度，形成持续世界坐标。
- 对手信标：在第三幕增加可见度，呼应统一匹配池。
- 航行核心：在第一幕与第四幕形成明确视觉焦点。

### 设计决策

**为什么不把 DOM 截图成纹理？**

实验 API 兼容性不足，传统 DOM 截图方案也会增加同步、字体与跨域问题。当前内容层保持原生 DOM，画布负责空间氛围即可达到目标。

**为什么不加载外部模型？**

程序化几何包体更稳定、没有版权与加载失败风险，也更容易形成原创视觉。

## 功能：四幕世界状态过渡

### 涉及文件

- `apps/web/components/product/HomeExperience.tsx`
- `apps/web/components/product/OrbitalWorld.tsx`

### 状态模型

每幕定义一组静态目标值：镜头位置、世界旋转、星球位置、主色、信标可见度和轨道亮度。渲染循环只把当前值插值到目标值，不创建新的场景对象。

| 幕 | 镜头与构图 | 环境重点 |
| --- | --- | --- |
| 0 接触 | 中近景，星球偏右 | 扫描核心、少量目标 |
| 1 循环 | 镜头拉远，轨道展开 | 三层轨道与航线 |
| 2 网络 | 构图横移，对手信标增强 | 节点与连接密度 |
| 3 上行 | 镜头重新收束 | 锁定核心与能量色温 |

场景切换使用基于帧时间的阻尼插值，不新增额外计时器。reduced-motion 时直接应用目标状态并只在状态变化或尺寸变化时渲染一次。

## 功能：性能、动效偏好与失败降级

### 生命周期

- 初始化失败时捕获错误并停止，不修改页面可用状态。
- 像素比桌面上限为 1.5，小屏上限为 1.25。
- 小屏减少粒子与信标数量。
- `document.hidden` 时停止动画帧，恢复可见后继续。
- resize 时更新 renderer 尺寸和相机宽高比。
- 卸载时取消动画帧、移除监听器，并释放 geometry、material 与 renderer。

### 错误策略

画布没有用户可恢复的业务状态，因此不显示错误提示；失败后保留 CSS 星场是更合适的降级。开发环境可以写入一次 `console.warn`，但不能重复刷屏。

## 功能：视觉层级与响应式整合

### 涉及文件

- `apps/web/app/game-ux.css`

增加画布固定层、场景色罩与边缘暗角。现有雷达和网络图降低不透明度，作为近景 HUD，而不是与三维世界争夺焦点。DOM 内容区域增加局部对比度保护，不使用整块不透明面板。

移动端保留星球与少量轨道，隐藏细小信标文字类装饰。reduced-motion 同时关闭 CSS 扫描动画和 Three.js 持续帧。

## 技术验收标准

- Three.js 不进入首页服务端模块初始化，也不阻塞 DOM 内容渲染。
- 画布不接收鼠标事件，不影响焦点顺序或链接点击。
- 四幕切换复用同一 renderer 和同一组场景对象。
- 后台标签页不持续请求动画帧，卸载后没有遗留监听器或 WebGL 资源。
- reduced-motion 只在状态/尺寸变化时渲染，不运行持续循环。
- WebGL 失败不影响现有 CSS 降级和全部操作。
- web test、lint、typecheck、build 通过。
