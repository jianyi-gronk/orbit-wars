# 任务

## 实施原则

- 第一阶段只交付公开互联网 2P 闭环；4P、赛季、赛事和自定义依赖后置。
- 人类与 Agent 复用同一 `ObservationV1`、`CommandBatchV1`、规则、匹配池和舰队 rating；控制方式只作标签。
- 排位比赛必须固定 `ruleset_id`、seed、槽位、控制者和策略版本，且能够由命令流确定性复现。
- 品牌、舰船、阵营、声音和页面构图使用原创资产；只借鉴太空歌剧氛围与时尚编辑设计方法。
- 每个任务都包含测试或可演示的完成条件；未满足依赖时不提前接入临时协议。

## 阶段 1：仓库与共享基础

- [x] **T01 初始化多包工作区与质量门禁** — 建立 `apps/web`、`services/api`、`services/match-worker`、`services/agent-sandbox`、`packages/contracts`、`packages/orbit-engine-py`、`packages/design-tokens` 和 `infra`，配置统一开发命令、格式化、静态检查、单测与 CI。依赖：无。完成条件：全新环境可安装依赖，空骨架的 lint、typecheck 和 test 在 CI 通过。
- [x] **T02 建立本地基础设施与配置规范** — 用 Docker Compose 启动 PostgreSQL、Redis 和 S3 兼容对象存储，定义本地/测试环境变量、健康检查、测试数据清理和密钥占位规则。依赖：T01。完成条件：一条命令可启动和停止依赖服务，API/Worker 可通过健康检查访问三类基础设施，仓库不包含真实密钥。
- [x] **T03 建立版本化共享契约生成链** — 以 Pydantic 定义 `ObservationV1`、`CommandBatchV1`、比赛消息、结果、参与者和回放基础类型，生成 JSON Schema 与 TypeScript 类型，并加入兼容性快照测试。依赖：T01。完成条件：Python 与 TypeScript 消费同一生成源，NaN、Infinity、错误 step 和越界命令等非法样例被一致拒绝。
- [x] **T04 建立数据库、身份验证与幂等写入底座** — 配置 SQLAlchemy/Alembic，创建用户、舰队、策略版本、比赛、参与者、命令、rating、rating event 和回放工件模型；接入 OIDC 会话验证、公开 ID 和通用 `Idempotency-Key` 中间件。依赖：T02、T03。完成条件：迁移可从空库正向执行和回滚，重复写请求返回同一业务结果，内部 UUID 不直接暴露为公开链接。

## 阶段 2：规则引擎与原创界面基础

- [x] **T05 提取并固定 Orbit Wars 2P 规则引擎** — 从 Apache-2.0 来源提取最小 Python 引擎包，保留 License/NOTICE，锁定第一版 `ruleset_id`、默认参数、动作适配和 state hash，不在生产运行时依赖 Kaggle master。依赖：T01、T03。完成条件：固定 seed 可 reset/step/snapshot，原始 `[[from_planet_id, angle, ships], ...]` 动作可无损转换，来源可审计。
- [x] **T06 建立确定性 golden 回归套件** — 选取现有 Orbit Wars 单测、固定 seed、历史 replay 和代表性 v69 对局，逐步比较 planets、fleets、reward、结束步和 state hash，并明确浮点容差与运行平台。依赖：T05。完成条件：同一输入重复运行全量一致，迁移引擎与 oracle 的选定样本通过逐步对比，差异输出能定位到具体 step 和字段。
- [x] **T07 实现 Match Worker 引擎适配与比赛状态机** — 实现 `queued → preparing → ready → running → finalizing → finished` 及 failed/forfeited/cancelled 分支，记录 ruleset、seed、槽位、命令日志和每步权威快照。依赖：T04、T05。完成条件：无控制器的固定脚本比赛可完整运行，非法状态转换被拒绝，平台故障与玩家失败不会混用状态。
- [x] **T08 建立原创设计 token 与组件基线** — 定义 editorial/tactical 两套密度、色彩、字体、轨道母题、阵营双编码、响应式和 reduced-motion token，建立原创/第三方资产来源清单，并实现少量可复用版式与 HUD 基元。依赖：T01。完成条件：Storybook 或组件预览覆盖品牌页与战术页两种场景，禁用动效后信息完整，资产清单可审计且无受保护影视元素。

## 阶段 3：舰队、策略版本与 Agent 安全链路

- [x] **T09 实现账号与单舰队领域 API** — 完成创建、读取、编辑舰队与“一账号一支活跃舰队”约束，校验舰队名、代号、宣言、策略倾向和原创外观描述，并生成不可枚举公开档案标识。依赖：T04。完成条件：所有权和内容边界测试通过，重复创建被稳定拒绝，公开响应不泄露私有字段。
- [x] **T10 实现不可变策略版本与对象存储** — 支持策略包、manifest、版本说明、来源、SHA-256 hash、运行镜像版本和 `uploaded/validating/ready/rejected` 生命周期；历史版本设为当前只移动指针。依赖：T02、T04、T09。完成条件：相同内容可去重，ready 后包内容不可修改，数据库与对象存储失败可安全补偿或重试。
- [x] **T11 构建固定依赖的 Agent Sandbox** — 实现非 root、无网络、只读根文件系统、tmpfs、CPU/内存/进程/日志/时间上限的容器和 JSONL runner，兼容 `def agent(obs)`。依赖：T01、T03。完成条件：正常策略可响应 observation；网络、宿主文件、环境凭据、fork bomb、超时和超量日志测试均被隔离或终止。
- [x] **T12 实现策略验证流水线与内置基线 Bot** — 对上传包执行安全解包、导入 smoke、契约、资源和固定局测试；接入基础 Bot、训练 Bot 与 v69 专家 Bot，并把验证结果写回版本状态。依赖：T06、T10、T11。完成条件：坏包得到可理解且不泄露内部信息的拒绝原因，ready 版本能在固定测试局运行，创建舰队可绑定一个立即可用的基础版本。
- [x] **T13 实现 Agent Key、Agent API 与 Guide** — 提供单舰队 scoped key 的创建、只显示一次、摘要存储、轮换、撤销和审计；实现 fleet/versions/opponents/matches 接口，并编写可被外部 AI 直接使用的规则、示例和错误码文档。依赖：T09、T10、T12。完成条件：scope 越权、撤销 key 和速率限制测试通过，Guide 中的最小示例可独立发布一个验证通过的版本。
- [x] **T14 实现模拟训练编排与资源限额** — 支持选择内置对手、公开可训练对手和地图发起 2P 模拟，使用不可变策略版本，记录结果但不创建 rating event，并按舰队和 key 限流。依赖：T07、T12、T13。完成条件：Agent API 与站内入口都能创建模拟，重试保持幂等，模拟失败/完成状态可查询且永不改排名。

## 阶段 4：统一控制与实时比赛

- [x] **T15 实现统一控制器、回合时钟与命令校验** — 建立 Human Adapter 和 Agent Adapter，共用同一步 Observation 与 CommandBatch；实现默认 3 秒窗口、每回合最多 6 条 launch、库存预算、空动作和稳定错误码。依赖：T03、T07。完成条件：两类控制器的契约一致性测试通过，迟到、重放、越权、超预算和错误 step 均被确定性拒绝，动作在截止前对对手不可见。
- [x] **T16 实现比赛创建、队列与短期票据** — 提供训练/排位比赛创建 API、随机槽位和服务端 seed，锁定 controller_type 与 strategy_version，建立 Redis 队列及短期 match ticket。依赖：T04、T09、T15。完成条件：重复请求不产生重复比赛，排位开始后不能切换控制者，票据仅能访问指定比赛与玩家槽位。
- [x] **T17 实现 WebSocket 网关与实时同步** — 完成 snapshot、turn open/accepted/closed、frame、finished、error 和 resync 消息，通过 Redis Streams/PubSub 连接 API 与 Worker，并校准服务端截止时间。依赖：T02、T15、T16。完成条件：两个客户端可同步完成脚本比赛，乱序/重复消息不破坏状态，短暂断线可从最后 step 续接。
- [x] **T18 实现人类战术指挥客户端** — 用 PixiJS 绘制权威战场，DOM HUD 支持选择源星、拖拽/选择方向、舰船数量、轨迹预览、待提交命令、倒计时和键盘操作；客户端不自行判定碰撞或胜负。依赖：T08、T17。完成条件：玩家可在时限内完成一个完整命令周期，服务端拒绝会明确回显，低性能模式与 reduced motion 下仍可操作。
- [x] **T19 接入 Agent 执行器与加速 Agent 对局** — Worker 为每个 Agent 启动独立 Sandbox，处理 requestId、1 秒 CPU 限额、overage、日志截断、崩溃和连续超时；Agent-v-Agent 可跳过墙钟等待但保留逻辑 step。依赖：T11、T12、T15、T16。完成条件：human-v-agent 与 agent-v-agent 均通过端到端脚本局，超时/崩溃能归因并判负，双方永远看不到对手当步动作。
- [x] **T20 实现断线、Worker 恢复与幂等终局** — 持久化 checkpoint、command log 和 state hash；人类连续错过默认 10 个窗口判负，Worker 崩溃可恢复，无法确定性恢复则 failed 不计分，finalizing 可安全重试。依赖：T07、T17、T19。完成条件：故障注入测试覆盖浏览器断线、Redis 短断、Worker 重启、对象存储失败和重复终局，恢复后的 state hash 与无故障运行一致。

## 阶段 5：回放、战斗解释与分享

- [x] **T21 实现流式回放写入与版本化存储** — 逐步写入权威帧和原始命令，每 20 步 checkpoint、其间 delta 编码，终局压缩上传并记录 checksum、schema、ruleset 和参与者归因。依赖：T03、T07、T20。完成条件：长局无需全量驻留内存，上传失败可重试，公开工件不含密钥、源码、会话票据和内部堆栈。
- [x] **T22 实现关键事件、指标与事实型胜因** — 从权威回放派生首次占领、母星易手、最大出击、产能/兵力领先反转、淘汰、超时/断线与终局事件，并计算星球、产能、驻守、在途和控制率曲线。依赖：T21。完成条件：至少五类事件可由 fixture 稳定复算，胜因只引用可验证指标，事件提取器升级不修改原始帧。
- [x] **T23 实现公开回放 API 与交互播放器** — 提供无需登录的公开链接、分段加载和短期 CDN/流式访问；播放器支持暂停、调速、逐步、拖动、checkpoint seek、事件跳转、双方标签与排名变化。依赖：T08、T21、T22。完成条件：桌面与移动端均能打开分享链接，任意 checkpoint seek 状态正确，移动端观战不要求 WebGL 高性能特效。

## 阶段 6：统一排位、匹配与公开档案

- [x] **T24 实现舰队统一 rating 与幂等结算** — 以 `fleet_id` 保存单组 `mu/sigma/display_score`，通过隔离的 RatingService 计算保守展示分和段位，事务内创建唯一 `rating_events.match_id`。依赖：T04、T20。完成条件：human 与 agent 结果更新同一 rating，重复 finalizing 只结算一次，training、failed 和不可归因错误不结算。
- [x] **T25 实现匹配、挑战和反刷分规则** — 优先同实力和低重复度匹配，限制悬殊分差定向挑战与短期重复对手，统一处理 human-human、human-agent 和 agent-agent，并记录可审计的匹配原因。依赖：T16、T24。完成条件：三种控制组合使用同一规则，刷同一弱对手会降低或停止分数变化，正常匹配不会被分析筛选拆成独立榜。
- [x] **T26 实现统一排行榜与公开舰队档案 API** — 提供今日、本周、历史视图和控制方式筛选标签，展示段位、战绩、近期变化、版本谱系、代表回放与每场 controller/strategy_version 归因。依赖：T09、T23、T24、T25。完成条件：任何筛选都不生成第二套 rating，匿名用户可读取公开档案，历史对局归因不随“当前版本”变化。

## 阶段 7：完整产品体验

- [x] **T27 实现品牌首页、账号入口与舰队创建闭环** — 用原创太空歌剧和时尚编辑式构图说明“亲自指挥或交给 Agent”，展示真实战斗入口，并让新用户创建舰队、获得基础策略后选择首场手动训练或 Agent 模拟。依赖：T08、T09、T12、T14、T18、T23。完成条件：未接触项目的测试者 30 秒内能复述双控制玩法，非开发者可在一次连续流程中完成首场训练或模拟。
- [x] **T28 实现舰队指挥中心与 Agent 管理体验** — 展示当前版本、统一排名、近期战绩、策略历史和控制标签，支持生成/撤销 key、发布状态、版本回滚指针，并把 Agent Guide 与示例接入界面。依赖：T13、T14、T24、T26。完成条件：用户可完成 key 全生命周期与历史版本切换，任何页面都不显示 key secret、私有源码或内部日志。
- [x] **T29 实现竞技场、模式选择和公开榜单页面** — 提供 human/agent 控制选择、训练/排位边界提示、匹配/挑战入口、不可撤销确认、队列状态、统一排行榜和编辑式舰队档案页面。依赖：T18、T19、T25、T26。完成条件：正式战前控制者和计分影响清晰可见，开赛后不能切换，比赛结束可直达永久战绩与回放。

## 阶段 8：上线质量、运维与验收

- [x] **T30 建立可观测性、安全与隐私门禁** — 贯通 request/match/step/sandbox trace，采集排队、回合延迟、迟到、资源、崩溃、上传、结算和重连指标；配置敏感日志过滤、限流、告警和数据保留规则。依赖：T13、T20、T21、T24。完成条件：能从一次失败比赛追踪到明确责任域，高优先级 determinism/重复结算/沙箱逃逸信号可告警，日志扫描不发现密钥或私有代码。
- [x] **T31 完成跨闭环自动化与故障验收** — 为“创建→手动训练”“创建→Agent 发布→模拟”“统一排位→结算→回放→档案”建立端到端测试，并加入幂等、恢复、权限、协议一致性和基础设施故障场景。依赖：T27、T28、T29、T30。完成条件：产品与技术验收标准均有自动化测试或明确的人工验收记录，三条核心闭环在干净环境稳定通过。
- [x] **T32 完成性能、无障碍、原创性与发布准备** — 验证战场 60 FPS/低性能 30 FPS、frame 预算、回放 seek、键盘与 reduced motion、移动端观战；执行 Star Wars/AgenTank/时尚参考相似性审查，准备生产配置、迁移、备份恢复、回滚和事故手册。依赖：T23、T27、T29、T31。完成条件：性能与可用性预算达标，原创资产审计通过，预发布环境完成一次部署、数据库迁移、备份恢复和应用回滚演练。

## 建议实施波次

- **波次 A：可复现底座** — T01–T08；交付可独立运行和验证的规则引擎、契约与界面基线。
- **波次 B：Agent 与模拟闭环** — T09–T14；交付“创建舰队→发布策略→运行模拟”。
- **波次 C：实时统一对战** — T15–T20；交付 human-human、human-agent、agent-agent 的权威比赛执行。
- **波次 D：回放与竞技产品** — T21–T29；交付解释、分享、统一排名和完整站内体验。
- **波次 E：公开上线** — T30–T32；完成安全、故障、性能、无障碍与原创性验收。
