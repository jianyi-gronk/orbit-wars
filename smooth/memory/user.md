# 用户记忆

从日常对话中沉淀的长期偏好和协作模式。

## 偏好
- 面向公开用户的网站默认需要同时支持简体中文与英文，不能只把中文作为临时占位文案。
- 游戏网站的首屏和登录后主页面必须给出单一、明确、可完成的开局入口；不能要求新用户从后台式导航中推断“创建角色/舰队 → 匹配 → 开战”的路径。
- Orbit/Wars 应同时构思并逐步开放 Human 手动参与和 Agent 自主对战；两者共用唯一 rating 与榜单，只标识控制方式。可以先用训练 Beta 验证手动可玩性，但不能把 Human 能力永久藏在默认入口之外。
- 竞技积分优先展示为用户易理解的“大段位 + 小段位 + 段内积分”，同时保留底层唯一总积分与全服名次。
- 游戏网站应优先呈现“正在进入一场游戏”的沉浸感，避免后台管理页气质；一级导航控制在少量核心入口，历史、指南、更新等收进次级菜单但仍可到达。
- Orbit/Wars 首页可以采用全屏分幕叙事：滚轮/触控推动上一幕退场与下一幕登场，但一个连续滑动手势最多切换一幕；关键点击提供短促、有反馈的游戏化动效，同时必须保留键盘操作、reduced-motion 降级和移动端无溢出。
- 代码协作中，每完成一个内聚改动就及时 commit；同一轮的多个 commit 在全部验证完成后统一 push，避免工作区混乱。
- 视觉可以参考 HTML-in-Canvas 与 Three.js 的沉浸感，但应保持原创，并避免把浏览器实验 API 作为公开网站的核心依赖。
- 用户提供明确视觉参考时，优先复现参考中的主体比例、层级与克制程度；不要擅自叠加虚线环、刻度、尖刺等会改变视觉语义的装饰。
- 中文大字号多行标题应保留清晰的垂直呼吸，不能用明显低于字形高度的行高制造层叠；海报感优先通过字号、字重、留白和局部描边实现，并需逐页检查中英文实际换行。
- Orbit/Wars 不应要求用户预先拥有外部 Agent 才能开始；平台模板应能直接参赛，并提供站内策略优化闭环，外部 Agent 接入作为可选高级入口。
- Orbit/Wars 全局页头应保持克制：Logo、竞技场、统一榜单、指挥中心和语言切换同排展示，避免任务菜单与重复“立即开始”按钮挤占主导航。
- Orbit/Wars 的登录应在当前任务上下文中以标准轻量弹窗完成，不使用独占整页的大型认证舞台，也不在弹窗堆叠产品介绍、功能卖点或授权长文案；可参考“韭见”的弹窗结构，但视觉保持原创且首发仍只展示 GitHub。
