进入前置调研模式。这个阶段用于在正式写 `product.md` 之前收集事实、现状、风险和开放问题，产出 `research.md`。

**默认语言：除命令、文件名、代码标识、引用原文外，面向用户的回复和生成的 Smooth 文档都用简体中文。**

**记忆意识：** 开始前按需读取 `smooth/memory/user.md`、`smooth/memory/pitfalls.md` 和相关 `smooth/memory/domains/<topic>.md`。本阶段若出现用户反驳、重复踩坑、可复用偏好，或能改进分析 / 代码 / 文档 / 流程生成的规则，先判断是否值得长期保留；值得时主动更新 memory 或本变更的 `pitfalls.md` / `lessons.md`，一次性细节只修正当前输出。

**对话输出：** 面向用户先说明当前判断或正在做的动作，再给关键产物摘要和下一步。不要默认整篇粘贴产物内容，除非用户要求或内容很短；需要用户决策时给出少量选项，并标出推荐默认。

**重要：前置调研是可选阶段。** 它帮助后续产品需求更准确，但不替代 `/smooth:product`。不要在这里定最终需求、做技术方案决策或实现代码。

**输入**：`/smooth:research` 后面的内容可以是调研主题、变更名或用户想探索的问题。可能是：
- 变更名：`tracking-events-v2`
- 问题：`先看看现在埋点链路是怎么跑的`
- 空输入：询问用户要调研什么

---

## 工作姿态

- **先看事实**：区分已验证事实、推断、观点和待确认问题。
- **代码库优先**：如果是项目内需求，先读现有代码、文档、配置和调用关系。
- **来源明确**：如果需要外部资料，记录来源、时间和不确定性。
- **不急着下结论**：调研不是拍板阶段，重点是降低产品讨论的盲区。
- **面向后续决策**：输出要能帮助 `/smooth:product` 更快形成清晰需求。

---

## 你要做什么

1. **确定调研名**

   如果用户没有提供明确名称，就从主题中推导 kebab-case 名称，并创建目录：

   ```bash
   mkdir -p smooth/changes/<name>
   ```

   如果目录已存在，先读取已有产物，尤其是 `research.md`、`workpad.md`、`product.md`。

2. **开展调研并写 `research.md`**

   持续更新 `smooth/changes/<name>/research.md`，标题使用：

   ```markdown
   # 前置调研
   ```

   推荐结构：

   ```markdown
   # 前置调研

   ## 调研问题
   - <这次想搞清楚什么>

   ## 已验证事实
   - <来自代码、文档、命令或可靠来源的事实>

   ## 当前现状
   - <系统、流程、用户路径或业务现状>

   ## 风险与约束
   - <可能影响需求判断的风险、依赖、限制>

   ## 开放问题
   - <仍需用户、代码或外部资料确认的问题>

   ## 对产品讨论的启发
   - <后续 /smooth:product 应重点讨论的方向>

   ## 来源与证据
   - <文件路径、命令、链接、日期、观察结果>
   ```

   如果调研范围较小，可以只保留必要章节。

3. **同步 `workpad.md`**

   如果调研中出现了计划、验证想法或长期疑问，更新 `workpad.md`。不要把 `workpad.md` 写成调研全文，它只是过程记录。

4. **建议下一步**

   调研结论足够支持需求讨论时，建议进入：

   - `/smooth:product`

   如果调研还缺关键信息，先说明缺口和下一步怎么查。

---

## 守则

- 不要把调研结论写成最终需求；最终需求属于 `product.md`。
- 不要做技术方案拍板；架构设计属于 `technical.md`。
- 不要写应用代码。
- 不要把未经验证的猜测当事实；必须标注为推断或待确认。
- 外部信息可能变化时，要说明日期和来源；需要实时性时应验证最新来源。
